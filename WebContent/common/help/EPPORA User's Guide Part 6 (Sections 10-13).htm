10.4 Measuring Job Performance Process

 

If all the project team members are building identical gizmos, than measuring performance would be simply tallying how many gizmos a project team member built on average per day.  The individuals who created the greatest number of gizmos would be the most productive.  Unfortunately there are very few, if any, projects that build identical gizmos.  Too often project team members feel that their productivity does not have a direct correlation with their remuneration.  Rather their relationship with their boss or their boss’s boss along with their ability to tout their accomplishments as opposed to their actual accomplishments is major influences in determining salary adjustments and bonuses.  It would be beneficial for nearly every organization conducting projects if there were an objective method for measuring an individual’s productivity.  This is exactly what EPPORA provides as described previously in Sections 5i through 5k.  The major purpose of this measurement is to improve an organization’s use of its precious labor resources, by assigning the most productive resources to the most important project tasks.  A side benefit of this is that management can now look at scientific data when assessing salary adjustments and bonuses.  Productivity isn’t the only factor in determining how well someone should be compensated, but it is one of the key if not the key factor.  An additional benefit is that if individuals believe their productivity is the major contributing factor to their financial success, they will tend to work hard and waste less time at work talking about non-work issues and events.

10.5 Estimating Project Costs

 

There are numerous ways different industries and organizations within an industry estimate project costs.  In the IT industry it is often how many lines of code in the application times some cost per line of code.  Another method involves the complexity of the application based on the depth of nesting.  In construction it might be how many square feet times some quality factor times other factors. Consider the following two statements:

 

1.     What a project is all about is its requirements.

2.     The cost of implementing a project is the cost of implementing all the project’s schedule tasks.

 

In EPPORA we allow the project’s Business Analyst, Project Manager or Project Portfolio Manager to link each low-level requirement to the schedule tasks needed to accomplish the requirement.  Each task that is linked to a requirement contains a specification as to what percentage of the task is used for the requirement it is being linked to.  Since each low level task has associated with it labor costs, inventory costs (if any) and other costs (if any), it is simple in EPPORA to calculate the cost of a project, the remaining cost of a project, the cost of a requirement (low-level or parent) or the remaining cost of a requirement.

12.         Scripting Processor

EPS uses IOPUS’s iMacro product to help support its testing activities.  iMacro is used by EPS’s software engineers and test engineers to assure that the EPPORA product satisfies each and every requirement. The execution and analysis of the test cases for a product the size of EPPORA is a timely, error-prone, expensive activity.  To complicate matters, nearly all software experts agree that when even a minor change is made to a software product, to verify the product prior to a release, one should re-run the entire test suite (to minimize regressions).  The EPPORA system is a complex, large application.  Since EPPORA is a tool providing critical prioritization, allocation of precious labor resources and costing data, it is imperative that the results EPPORA provides are valid and new releases can be verified in a timely manner. 

The cost of implementing the initial test suite with iMacro was slightly more expensive than testing EPPORA by a set test engineers running the test procedures (estimated at 10-20% more).  The payoff in creating tests using iMacro occurs when the tests are run more than once.  The cost for subsequent testing is a small fraction (less than 1%) of the initial test development cost.  In addition, with iMacro, EPPORA is able to test new releases in a small fraction of the time it would have taken using multiple test engineers.   It is also easy to enhance the testing of EPPORA as new features are added or latent defects occur.

iMacro is used in its “record mode” to specify tests to be executed by EPPORA.   iMacro simplifies the task of producing executable actions that are repeatable by the EPPORA system. iMacro is independent of the EPPORA platform or Internet browser.  iMacro can emulate any EPPORA keystroke or mouse click a user can perform by processing a text file. The text file contains a series of scripting language commands that instruct EPPORA to perform various actions, such as selecting an object, pressing a radio button, entering characters into a text edit object, selecting a check box, etc. In addition, the text-scripting file can include references to other files within it that perform additional actions.  iMacro can take a snapshot of the screen and create a stylized text file representation of the page.  iMacro can also compare two text files and report any differences.  iMacro keeps a log that lists all errors encountered, as well as other user supplied and script supplied information.

13.         Automated End-of-Business Activities

EPPORA performs behind the scene activities even when there are no active users. The activities take place at the end of the business day and take less than an hour of wall clock time. The default End-of-Business time is 10 PM.  The indicators enclosed in parentheses PPM (Project Portfolio Manager), PM (Project Manager), BA (Business Analyst), SP (Sponsor), ADM (Administrator), EXEC (Executive) and USER (Designated User) are used to indicate who receives messages when an issue occurs.  A “/P” signifies that only this user type for the specified project receives the message.  Activities include:

         Daily

             Initialization

1     Delete each user’s previous task assignments.

2     Delete each user’s previous messages.

Project Ranking

1.    If any change in the weight of a criterion or a new criterion is added, then compute each project’s ranking score (PPMs and EXECs).

2.    If a project’s score for a criterion has changed, then compute the project’s ranking score (PPMs and EXECs).

Labor Categories

1     Verify that at least one labor category exist (PPMs).

2     Verify when new users belong to a labor category or a user no longer supports a labor category or a user that belongs to a labor category has a change in hourly salary then the average, highest and lowest hourly salary are computed for the labor category (PPMs) .

3     Verify that the labor categories’ ”Project Manager,” “Business Analyst” and “Project Portfolio Manager” exist (PPMs).

Divisions

1     Verify that at least one division exists (ADMs).

2     Verify each existing division has at least one user assigned to it (ADMs).

3     Compute the exchange rate for each division for the coming day.

Users

1     Verify that each user has a supervisor specified, except if the user has a job title of CEO, President, Managing Director or Owner (PPMs).

2     Verify that each user has one or more labor categories specified (PPMs and USER).

3     Verify that each user has a work telephone number specified (PPMs and USER).

4     Verify that each user has an email address specified (PPMs and USER).
